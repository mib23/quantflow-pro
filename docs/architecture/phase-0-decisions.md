# Phase 0 方案决策

## 一期业务边界

- 交易市场：仅覆盖美股单市场。
- 账户模型：单用户单交易账户。
- Broker：一期固定 `Alpaca`，二期预留 `IBKR` 扩展位。
- 数据源：一期以 `Alpaca Market Data + Broker API` 为主，补充自维护订单、风控和审计数据。
- 必须打通的闭环：`登录 -> 账户总览 -> 行情 -> 下单 -> 订单状态 -> 风控`。

## 用户角色

| 角色 | 核心职责 | 一期权限范围 |
| --- | --- | --- |
| 管理员 | 系统配置、用户与环境管理、审计查看 | 全量配置与只读运营视图 |
| 交易员 | 账户查看、下单撤单、订单跟踪 | 交易执行、账户与订单查询 |
| 研究员 | 研究、回测、策略草稿维护 | 研究视图和只读风控视图 |

## 一期明确不做

- 多租户
- 多 Broker 并行接入
- 复杂审批流
- 参数优化与高级研究任务编排
- 生产级微服务拆分

## 工程决策

### 后端

- 服务形态：模块化单体。
- Web 框架：`FastAPI`
- 数据库：`PostgreSQL`
- 缓存与队列：`Redis`
- ORM：`SQLAlchemy`
- 迁移：`Alembic`
- 后台任务：`RQ`

### 选择 RQ 的原因

- Phase 0 目标是快速形成最小可运行骨架，RQ 对 Redis 依赖单一，配置成本更低。
- 一期后台任务以回测排队、同步补偿、异步通知为主，RQ 足以覆盖。
- 后续若出现复杂编排，可在不破坏模块边界的前提下升级到 Celery。

### 前端

- 基座：`React + TypeScript + Vite`
- 路由：`React Router`
- 服务端状态：`TanStack Query`
- 请求层：`Axios`
- 本地状态：`Zustand`
- 表单与校验：`React Hook Form + Zod`

## 环境划分

| 环境 | 用途 | 关键配置规则 |
| --- | --- | --- |
| `local` | 本地开发 | 允许 mock 与 debug，使用本地 PostgreSQL/Redis |
| `test` | 联调与自动化测试 | 固定种子数据、对接 paper trading |
| `prod` | 生产运行 | 禁用 mock，所有凭证走密钥管理 |

## 配置约定

- 前端环境变量统一使用 `VITE_` 前缀。
- 后端环境变量统一使用 `QF_` 前缀。
- OpenAPI 作为 REST 契约单一事实来源。
- Broker 原始结构不得直接暴露给前端，必须先映射为内部 DTO。
