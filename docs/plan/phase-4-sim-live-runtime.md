# Phase 4：仿真与实盘运行

## 阶段目标

- 在研究态回测闭环完成后，建立策略部署、启动、停止、监控与审计能力。
- 先打通仿真环境长期运行，再在同一模型下补齐实盘发布审批和运行保护。
- 将系统从“能研究和回测”推进到“能持续运行并被运维和风控观察”。

## 进入条件与完成定义

### 进入条件

- Phase 3 已提供稳定的策略版本、参数快照、回测任务与结果模型。
- Phase 1 的账户、订单、行情与 Phase 2 的风险、审计能力可被运行态复用。
- 后台 worker 具备执行异步任务的稳定基础，且 Broker paper trading 环境可用。

### 完成定义

- 用户可选择策略版本部署到仿真环境，创建运行实例并执行启动、停止、重启。
- 运行实例具备状态可见性、心跳、运行日志、异常告警和最小恢复手段。
- 实盘发布必须经过审批和审计，未通过审批不得进入 live 运行。
- 运行时产生的订单、日志、风险事件与实例 ID 关联可追踪。
- Phase 4 的重点是“可持续运行与可监控”，不新增研究功能面。

### 输出给下一阶段

- 稳定的运行实例模型、审批模型、日志模型和告警输入。
- 可被运营和报表阶段复用的实例状态、运行日志、实盘审计与告警数据。
- 仿真和实盘环境的部署与运行规范，为上线前稳定性和观测建设提供基础。

## 范围与非目标

### 本阶段必做闭环

- 运行实例管理：部署、启动、停止、重启、查询、心跳。
- 仿真环境运行：paper trading 账户、行情消费、订单回路、状态监控。
- 实盘审批：发布前校验、审批通过后发布、审计留痕。
- 日志与告警：策略日志、系统日志、异常快照、关键事件通知。
- 异常恢复：worker 崩溃、心跳丢失、Broker 异常时的最小恢复和人工接管。

### 可延后项

- 蓝绿发布、分批发布、多实例编排、自动扩缩容。
- 高级调度策略、资源配额管理、跨区域容灾。
- 复杂审批链、多级签核、自动回滚编排。

### 不在本阶段处理

- 报表中心、通知中心全面化。
- 大规模运营看板和最终上线演练。
- 高级研究特性和参数优化能力。

## 工作流拆解

### 前端

- 为策略增加部署入口，允许用户选择版本、环境、账户、参数快照并创建运行实例。
- 增加运行实例列表和详情页，展示实例状态、心跳时间、运行参数、关联账户、最近日志和最近告警。
- 增加启动、停止、重启操作，并对不同状态给出清晰禁用态和确认提示。
- 实盘部署入口必须带审批状态展示，未通过审批时禁止进入 live。
- 在页面中联动展示与实例相关的订单、风险事件和系统告警。

### 后端

- 建立 `runtime` 模块，定义运行实例生命周期、状态机和环境区分。
- 创建运行实例时固化策略版本、参数快照、目标账户、环境类型、提交人、审批状态。
- worker 负责拉起策略运行进程或任务循环，周期性上报心跳和健康状态。
- 将运行产生的下单、成交、风控事件与 `runtime_instance_id` 关联，便于审计和回放。
- 对 live 环境增加发布前校验：策略版本存在、回测结果合格、目标账户有效、审批通过、关键风控规则已启用。
- 对异常退出、心跳超时、Broker 连续失败设置实例状态迁移和告警触发规则。

### 数据与集成

- 落地 `runtime_instances`：实例 ID、策略版本 ID、账户 ID、环境、状态、启动时间、停止时间、最近心跳、错误摘要。
- 为审批流落地最小审批实体或审计扩展字段，至少记录申请人、审批人、审批结果、审批时间和备注。
- 运行日志建议采用结构化日志，按实例 ID、策略 ID、账户 ID、环境、级别索引。
- 实例状态机至少包含：`CREATED`、`STARTING`、`RUNNING`、`STOPPING`、`STOPPED`、`FAILED`、`DEGRADED`。
- 保持运行态仍通过 `BrokerAdapter` 访问 Broker，不允许策略代码直接耦合第三方 SDK。

### 运行与恢复

- 为仿真环境定义最小运行守护：心跳上报、订单回报消费、异常重试、可人工停止。
- 为实盘环境定义更严格限制：审批通过后才能启动，失败重试次数受限，异常时优先进入保护状态。
- 若 worker 崩溃或心跳丢失，系统先标记 `DEGRADED`，再按策略决定自动重启或人工接管。
- 对 Broker 上游异常做分类，如认证失败、限流、订单拒绝、行情中断，并记录恢复建议。

### 测试与运维

- 在 test 环境先完成 paper trading 长时间运行验证，再进入 live 审批链路联调。
- 记录实例级指标：运行时长、心跳延迟、订单失败率、重启次数、Broker 错误数。
- 明确实例停止、重启、强制终止的操作说明，避免误操作。
- 对运行配置和账户权限做最小隔离，避免研究员直接发起 live 运行。

## 关键接口、模型与事件

### REST API

- `GET /api/v1/runtime/instances`
- `POST /api/v1/runtime/instances`
- `GET /api/v1/runtime/instances/{instance_id}`
- `POST /api/v1/runtime/instances/{instance_id}/start`
- `POST /api/v1/runtime/instances/{instance_id}/stop`
- `POST /api/v1/runtime/instances/{instance_id}/restart`
- `POST /api/v1/runtime/deployments/{instance_id}/approve`
- `POST /api/v1/runtime/deployments/{instance_id}/reject`
- `GET /api/v1/runtime/instances/{instance_id}/logs`

### WebSocket 事件

- `runtime.status.<instance_id>`：实例状态和心跳变化。
- `runtime.logs.<instance_id>`：结构化日志摘要或最新日志片段。
- `risk.events.<account_id>`：继续作为运行期风险联动输入。
- `orders.status.<account_id>`：继续作为运行期订单联动输入。

### 核心数据模型

- `RuntimeInstance`：实例 ID、策略版本 ID、账户 ID、环境、状态、审批状态、最近心跳。
- `DeploymentApproval`：申请人、审批人、审批结果、备注、时间。
- `RuntimeLogEntry`：实例 ID、级别、来源、消息、时间、上下文摘要。

## 里程碑与依赖

### 里程碑 1：仿真部署与实例管理

- 完成实例创建、启动、停止、状态查询和列表展示。
- 产出：仿真环境最小运行闭环。

### 里程碑 2：运行监控与日志

- 完成心跳、日志、异常状态迁移和告警联动。
- 产出：实例可监控、异常可定位。

### 里程碑 3：实盘审批与保护

- 完成 live 发布审批、发布前校验和审计记录。
- 产出：受控的实盘发布路径。

### 关键依赖

- 依赖 Phase 3 提供可部署的策略版本与参数快照。
- 依赖 Phase 1、Phase 2 提供账户、订单、风险和审计共用模型。
- 如 live 联调条件不足，可先在 test 环境完成完整审批流模拟，但接口和状态机不可缺位。

## 验收与测试场景

### 测试矩阵

| 类型 | 必测内容 |
| --- | --- |
| 单元测试 | 实例状态机、审批状态流转、心跳超时判断、异常分类映射 |
| 集成测试 | 实例创建、启动、停止、日志查询、审批通过/拒绝、实例与订单关联 |
| 联调 / E2E | 从策略版本部署到仿真运行，查看日志和状态，执行停止或重启 |
| 人工验收 | 心跳中断触发告警、审批未通过无法 live、异常退出后进入保护状态 |

### 核心验收场景

1. 用户选择一个已通过回测的策略版本部署到 paper account，系统创建实例并进入 `RUNNING`。
2. 页面可看到实例状态、最近心跳、最近日志和关联订单。
3. 模拟 worker 中断或心跳超时，实例进入 `DEGRADED` 或 `FAILED`，并触发告警。
4. 用户提交 live 部署申请，审批未通过前无法启动实例；审批通过后才可进入 live 运行。
5. 实例运行期间产生的订单、风控事件和日志均可按实例追踪查询。

## 风险与回退方案

- 运行态与研究态耦合过深：必须坚持“版本快照输入、实例独立运行”的边界，避免直接执行草稿策略。
- 心跳和日志链路不稳定：实例状态判断优先依赖显式心跳，日志展示可延迟但状态不可失真。
- 实盘审批绕过风险：live 启动接口必须二次校验审批状态和用户权限。
- Broker 上游波动导致实例反复失败：设置重试上限和熔断保护，必要时自动停机并人工介入。
- 如自动恢复策略尚未稳定，可默认只告警不自动重启，优先保证实盘安全。

## 交付清单

- 运行实例、审批、日志与状态模型。
- 仿真部署、启动、停止、重启闭环。
- live 发布审批与发布前校验。
- 心跳、告警、异常恢复和实例级审计。
- 运行联调说明、测试脚本和故障演练用例。
