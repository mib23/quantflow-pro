# QuantFlow Pro 全栈功能改造计划

## 1. 目标与范围

本计划用于将当前项目从“交易系统前端原型”升级为“可上线、可运维、可审计、可扩展”的完整量化交易平台，覆盖以下范围：

- 前端业务系统完整化：仪表盘、交易、策略研究、风控、告警、账户、运营后台。
- 后端服务完整化：认证鉴权、行情接入、订单路由、持仓与账户、策略执行、回测、风控、通知、审计、报表。
- 数据体系完整化：市场数据、订单成交、账户资产、策略结果、风控事件、日志、指标与报表。
- 工程体系完整化：测试、CI/CD、环境管理、监控告警、安全基线、部署与运维。

## 2. 当前项目现状判断

### 2.1 已有能力

- 已具备 4 个核心业务页面雏形：`Dashboard`、`Trade`、`StrategyLab`、`Risk`。
- UI 视觉风格、信息布局和交易工作台交互方向已经较清晰，适合作为业务原型基础继续演进。
- 领域对象已有初步抽象，包括持仓、策略、订单、日志、盘口等基础类型。

### 2.2 主要问题

- 当前项目只有前端，尚无后端工程、数据库、缓存、消息系统和任务执行体系。
- 页面数据几乎全部来自 mock 常量，尚未接入 API、WebSocket、Broker、行情源和策略引擎。
- 缺少路由、用户体系、权限模型、表单校验、错误处理、全局状态管理和数据请求层。
- 缺少订单全生命周期、策略发布流程、回测任务管理、风控事件闭环和审计留痕。
- 缺少测试、日志规范、监控、部署方案和环境配置体系。

### 2.3 结论

项目当前更接近“高保真前端概念验证版”，不建议在现有代码上直接堆功能。建议采用“保留现有页面信息架构与视觉语言，重建工程骨架和后端体系”的方式推进。

## 3. 改造目标

### 3.1 业务目标

- 支持用户登录、账户管理、交易账户绑定。
- 支持实时行情、盘口、持仓、订单、成交、资产展示。
- 支持手工下单、改单、撤单、批量撤单、风控前置校验。
- 支持策略创建、版本管理、回测、参数优化、仿真、实盘部署、运行监控。
- 支持全局风控、账户风控、策略风控、标的风控和自动熔断。
- 支持消息通知、事件审计、报表导出和运行日志追踪。

### 3.2 技术目标

- 前后端分层清晰，接口契约稳定。
- 实时场景采用 WebSocket / SSE + 事件驱动架构。
- 回测、仿真、实盘执行解耦，支持异步任务和多进程扩展。
- 支持单机开发、测试环境、生产环境的标准化部署。
- 具备基础安全、可观测性和自动化测试能力。

## 4. 推荐目标架构

### 4.1 前端架构

- 技术栈建议：React + TypeScript + Vite。
- 补齐基础设施：React Router、TanStack Query、Zustand 或 Redux Toolkit、React Hook Form、Zod、Axios。
- 样式体系建议：将当前 CDN Tailwind 切换为工程化 Tailwind 配置，建立设计令牌、主题变量和组件规范。
- 图表与交易组件继续保留现有方向，但需抽象为可复用组件库。

### 4.2 后端架构

- 推荐采用单体优先、模块化分层的服务架构，第一阶段避免过早微服务化。
- 已确认后端技术路线：`FastAPI + PostgreSQL + Redis`
- ORM 与数据访问层建议采用：`SQLAlchemy + Alembic`
- 异步任务与后台作业建议采用：`Celery` 或 `RQ`
- 该路线与 Python 量化生态兼容性更好，便于接入回测、策略执行和数值计算能力。

### 4.3 核心基础设施

- 数据库：PostgreSQL
- 缓存：Redis
- 消息与任务：Redis Queue / RabbitMQ（二期可升级）
- 实时通信：WebSocket
- 对象存储：MinIO / S3
- 监控：Prometheus + Grafana
- 日志：Loki / ELK
- 部署：Docker Compose 起步，Kubernetes 作为后续演进

## 5. 业务域拆分

### 5.1 认证与用户域

- 用户注册、登录、刷新令牌、注销
- RBAC 权限模型
- MFA（二期）
- API Token 管理
- 用户偏好、主题、时区、通知配置

### 5.2 账户与经纪商接入域

- 交易账户绑定
- 多 Broker 适配层
- 一期推荐接入：`Alpaca`，用于快速完成美股模拟盘与实盘 API 闭环
- 二期扩展接入：`IBKR`，用于增强券商兼容性与更复杂账户场景支持
- Broker 接入统一通过 `BrokerAdapter` 抽象层实现，业务层不得直接耦合单一券商 SDK
- 账户资产、购买力、保证金、手续费
- 持仓、成交、对账与日终同步

### 5.3 行情与市场数据域

- 实时行情、Level 1 / Level 2
- K 线、分时、成交明细
- 标的主数据管理
- Watchlist、自选分组、搜索
- 数据质量校验与重连机制

### 5.4 订单与交易执行域

- 下单、改单、撤单、批量撤单
- 市价单、限价单、止损单、止盈止损单
- 订单状态机与成交回报
- 幂等请求、重试、失败补偿
- 交易前风控拦截

### 5.5 策略研究与回测域

- 策略文件管理
- 策略版本与参数模板
- 回测任务提交、排队、执行、结果存档
- 指标分析、权益曲线、交易明细、报告下载
- 参数扫描、Walk-forward、基准对比（二期）

### 5.6 实盘与仿真运行域

- 仿真账户运行
- 实盘部署审批
- 运行实例生命周期管理
- 策略日志、心跳、异常恢复
- 运行时参数热更新（有限范围）

### 5.7 风控与合规域

- 全局风控：日亏损、总敞口、回撤阈值、熔断
- 账户风控：资金占用、保证金、单账户限额
- 策略风控：单策略最大回撤、连续亏损限制
- 标的风控：黑名单、白名单、交易时段限制
- 审计与审批：关键操作审计、变更审批、事件回放

### 5.8 通知与运营域

- 站内通知
- 邮件、短信、Webhook、企业微信/钉钉
- 告警中心
- 运营报表、策略日报、风控日报

## 7. 后端建设计划

### 7.1 服务模块

- `auth`：登录、令牌、权限、会话
- `users`：用户与配置
- `accounts`：账户、资产、持仓、对账
- `market-data`：行情拉取、订阅、缓存、广播
- `orders`：订单生命周期管理
- `executions`：成交回报与状态同步
- `strategies`：策略元数据、版本、配置
- `backtests`：回测任务与结果
- `runtime`：仿真/实盘运行实例
- `risk`：规则引擎、预检查、熔断
- `notifications`：消息发送与订阅
- `audit`：审计日志与事件查询

### 7.2 API 设计原则

- REST 负责配置类与查询类接口
- WebSocket 负责行情、订单状态、风险事件、策略日志等实时推送
- 所有写操作带幂等键
- 所有关键资源带版本号和审计信息

### 7.3 数据模型建议

核心表建议至少包括：

- users
- roles / permissions / user_roles
- broker_accounts
- account_balances
- positions
- orders
- executions
- instruments
- market_ticks / bars
- strategies
- strategy_versions
- backtest_jobs
- backtest_results
- runtime_instances
- risk_rules
- risk_events
- notifications
- audit_logs

### 7.4 策略执行与回测

- 将“策略编辑”与“策略执行”解耦
- 回测任务异步执行，结果入库存档
- 实盘策略以独立 worker 运行，主服务仅做调度与监控
- 统一策略 SDK、日志接口、事件模型和风控钩子

## 8. 关键非功能需求

### 8.1 安全

- JWT + Refresh Token
- 密码加密、敏感字段脱敏、密钥分环境管理
- 限流、登录失败保护、审计日志不可随意删除
- Broker 凭证单独加密存储

### 8.2 性能

- 高频读场景优先缓存
- 行情与订单推送分频道
- 图表数据分页/聚合
- 回测任务独立资源池

### 8.3 可观测性

- 请求日志、领域事件日志、任务日志
- Trace ID 全链路贯通
- API SLA、任务成功率、推送延迟、订单失败率监控

### 8.4 可测试性

- 单元测试：领域逻辑、风控规则、订单状态机
- 集成测试：API、数据库、缓存、消息、Broker 适配层
- E2E 测试：登录、下单、回测、部署、风控触发

## 9. 分阶段实施路线

### Phase 0：方案确认与基础重构（1 周）

- 明确业务边界、角色模型、目标 Broker、数据源和部署环境
- 决定后端技术栈与数据库方案
- 输出接口规范、ER 图、领域边界图
- 前端完成目录重构、路由接入、状态层和请求层基建

详细任务清单：

1. 业务与范围确认

- 明确一期范围只覆盖美股、单市场、单账户、单 Broker。
- 确认一期 Broker 为 `Alpaca`，二期预留 `IBKR` 扩展位。
- 明确用户角色：管理员、交易员、研究员。
- 明确一期必须打通的业务闭环：登录 -> 账户 -> 行情 -> 下单 -> 订单状态 -> 风控。
- 明确一期不做的功能：多租户、多 Broker 并行、复杂审批流、参数优化、生产级微服务拆分。

2. 目标架构与工程决策

- 固化后端技术路线：`FastAPI + PostgreSQL + Redis`。
- 确认 ORM 与迁移方案：`SQLAlchemy + Alembic`。
- 确认异步任务方案：`Celery` 或 `RQ`，并统一一期选择。
- 确认前端工程基座：`React + TypeScript + Vite + React Router + TanStack Query`。
- 确认前后端同仓结构：前端保留现有目录，新增 `server` 目录承载后端。
- 确认环境划分：local、test、prod，定义各环境的配置管理规则。

3. 领域建模与模块边界

- 输出核心领域清单：auth、users、accounts、market-data、orders、strategies、risk、audit。
- 明确每个领域的职责边界、输入输出、依赖关系。
- 梳理核心实体与聚合：用户、账户、标的、订单、成交、持仓、风险规则、风险事件。
- 确定 `BrokerAdapter` 抽象接口边界，约束账户查询、持仓同步、下单、撤单、订单回报订阅能力。
- 明确哪些数据来自 Broker，哪些数据由本系统持久化维护。

4. 数据库与缓存设计

- 绘制一期 ER 图，覆盖 `users`、`broker_accounts`、`account_balances`、`positions`、`orders`、`executions`、`risk_rules`、`risk_events`、`audit_logs`。
- 定义关键字段、主外键、唯一约束、索引策略和审计字段。
- 明确订单状态机、订单幂等键、外部 Broker 订单号映射方案。
- 设计 Redis 用途：会话缓存、实时行情缓存、WebSocket 订阅通道、任务队列。
- 确定哪些表由实时同步写入，哪些表由异步任务补偿更新。

5. API 与事件契约设计

- 输出一期 REST API 清单，至少包括认证、账户、持仓、订单、成交、风险规则、风险事件。
- 输出一期 WebSocket 事件清单，至少包括行情推送、订单状态变更、风险事件、系统通知。
- 统一接口返回格式、错误码、分页格式、鉴权方式、时间与金额字段规范。
- 定义下单、撤单接口的请求体和响应体，并设计幂等机制。
- 定义前端查询模型与后端 DTO，避免页面直接依赖 Broker 原始结构。

6. 后端基础骨架搭建

- 创建 `server` 目录与 FastAPI 基础工程。
- 建立基础目录结构：`api`、`core`、`modules`、`integrations`、`workers`、`tests`。
- 接入配置管理、日志模块、异常处理、中间件、健康检查接口。
- 接入 PostgreSQL 连接、Redis 连接、Alembic 迁移框架。
- 建立 `auth`、`accounts`、`orders`、`market_data`、`risk` 模块空骨架。
- 创建 `integrations/brokers/alpaca` 目录，定义接口与占位实现。

7. 前端基础重构

- 将当前基于 `activeTab` 的页面切换重构为路由模式。
- 建立 `app`、`features`、`entities`、`shared`、`widgets` 的目录结构。
- 引入 API Client、TanStack Query、全局错误处理、统一加载态与空态组件。
- 把 mock 常量与 UI 组件解耦，改造成可切换 mock/api 的数据访问层。
- 为登录、账户总览、订单列表、下单面板预留真实接口接入点。
- 补充基础表单校验、金额/数量格式化和枚举字典工具。

8. 开发环境与协作规范

- 统一 `.env` 变量命名规则，区分前端与后端配置。
- 制定接口文档维护方式，建议使用 OpenAPI 作为单一事实来源。
- 统一代码规范、提交规范、分支策略和目录命名规范。
- 补充本地启动说明，保证前后端与数据库/Redis 可一键启动。
- 明确 mock 数据仅用于 UI 占位，不再作为业务事实来源。

9. 测试与验收准备

- 为后端接入最小可用测试框架，至少覆盖 health check、配置加载、数据库连接。
- 为前端接入最小可用测试框架或保留后续接入方案说明。
- 明确 Phase 1 联调前置条件和所需种子数据。
- 制定 Phase 0 评审清单，确保架构、ER 图、接口和目录骨架经过确认。

完成标志：

- 技术路线、Broker 方案、领域边界、环境划分已确认。
- `server` 后端骨架已创建，能够本地启动 FastAPI。
- PostgreSQL、Redis、Alembic 已接入并完成首个迁移初始化。
- 前端已切换到路由化结构，并具备统一请求层与状态管理基座。

## 12. 风险与注意事项

- 若先做页面联调而不先定领域模型，后续会频繁返工。
- 若过早做微服务拆分，会显著增加开发和运维复杂度。
- 若回测与实盘共用不清晰的代码路径，容易引入行为偏差。
- 若缺少审计与风控留痕，后续无法满足交易系统合规要求。
- 若直接在当前 mock 结构上扩展，技术债会快速放大。

## 13. 建议的立即行动项

1. 确认目标用户角色、交易市场、Broker 和行情源。
2. 按已确认技术路线细化后端基座：`FastAPI + PostgreSQL + Redis`。
3. 先产出领域模型、接口清单、数据库草图。
4. 先完成前端工程化重构，再开始逐页接 API。
5. 优先打通“登录 -> 账户 -> 行情 -> 下单 -> 订单状态 -> 风控”最小闭环。

## 14. 本项目的推荐实施策略

结合当前仓库规模，建议采用以下策略：

- 保留现有 UI 页面作为视觉原型与需求讨论基础。
- 将现有 mock 数据层全部替换为领域 API 和实时订阅模型。
- 在仓库内新增 `server` 目录，形成前后端同仓开发模式。
- 第一阶段只做单账户、单 Broker、单市场，先把闭环跑通。
- 第一阶段 Broker 明确选择 `Alpaca`，优先接入 paper trading、账户、持仓、下单、撤单和订单回报。
- 架构层面预留 `BrokerAdapter`，确保二期可以平滑扩展 `IBKR`。
- 回测与策略执行优先基于 Python 生态建设，后续再扩展多语言能力。

---

如果按“最快形成可演示且可扩展的版本”推进，建议先完成 `Phase 0 + Phase 1 + Phase 2`，这样即可得到一个具备认证、交易闭环、实时数据和基础风控的可用版本，再进入策略研究和实盘部署阶段。
