# Phase 5：运营、报表、稳定性与上线

## 阶段目标

- 在核心交易、风控、研究、运行能力具备后，完成上线前所需的运营、报表、测试、可观测性和部署收口。
- 让系统达到“可测试、可部署、可观测、可审计”的上线准备状态。
- 将前四阶段沉淀的数据和事件转化为运营可读、运维可用、审计可追溯的最终交付能力。

## 进入条件与完成定义

### 进入条件

- Phase 1 到 Phase 4 已提供稳定的账户、订单、风控、回测、运行实例和审计链路。
- Docker Compose、本地和 test 环境可重复启动，数据库迁移和基础依赖可用。
- 关键业务流已具备最小可运行能力，剩余工作聚焦在工程完备性和上线准备。

### 完成定义

- 系统具备通知中心、报表导出、审计查询等运营基础能力。
- 核心流程具备自动化测试，关键指标和异常可观测。
- 可在 test 和 prod 环境按标准步骤部署，并具备健康检查、回滚和变更记录。
- 完成基本性能压测、安全检查和敏感配置治理。
- Phase 5 不新增核心交易能力，而是收口质量、稳定性和上线流程。

### 输出结果

- 一套可执行的测试、发布、监控、审计和运营方案。
- 面向管理员、交易员、研究员的运营入口和报表能力。
- 上线前检查清单、发布流程和回退流程。

## 范围与非目标

### 本阶段必做闭环

- 通知中心：站内通知、Webhook 或邮件占位、关键告警收敛。
- 报表与查询：账户日报、策略日报、风控日报、审计查询与导出。
- 自动化测试：关键 API、关键交易路径、回测链路、风险拦截链路、运行实例链路。
- 可观测性：请求日志、任务日志、指标、Trace ID、监控看板。
- 部署与上线：Docker 化、环境配置、迁移执行顺序、健康检查、回滚说明。
- 稳定性与安全：压测、限流、敏感信息保护、Broker 凭证治理。

### 可延后项

- 多地域容灾、跨集群高可用、Kubernetes 正式生产编排。
- 高级 BI 报表、自定义仪表盘、长期存档策略自动分层。
- 短信、IM、多渠道通知编排。

### 不在本阶段处理

- 新的核心交易产品能力。
- 新的策略研究范式或回测引擎升级。
- 多 Broker 和多租户能力扩展。

## 工作流拆解

### 前端

- 增加通知中心页面，聚合风险告警、运行异常、系统通知和重要任务结果。
- 增加报表查询与导出入口，支持按日期、账户、策略、事件类型筛选。
- 增加审计日志页面或运营后台视图，支持查询关键操作和变更记录。
- 为关键页面补充错误页、空态、加载态和系统异常提示，确保线上可诊断。
- 若存在管理员视图，统一通过角色控制访问，不向交易员或研究员暴露高权限入口。

### 后端

- 建立 `notifications` 模块，统一通知写入、读取、已读状态和投递结果。
- 建立报表导出接口，覆盖账户日报、策略日报、风控日报、审计导出。
- 为前四阶段的关键链路补齐 Trace ID 透传：HTTP、WebSocket、队列任务、运行日志。
- 暴露系统指标接口或埋点，覆盖 API 延迟、错误率、任务成功率、推送延迟、订单失败率、实例心跳延迟。
- 为生产环境增加限流、中间件防护、敏感配置校验和安全头等最小基线。

### 数据与报表

- 设计通知实体，至少包含通知类型、来源、级别、目标用户、已读状态、投递结果。
- 报表优先复用前四阶段沉淀的数据表与审计表，不引入新的业务事实源。
- 账户日报至少包含账户净值、收益、持仓摘要、订单成交摘要。
- 策略日报至少包含运行实例摘要、收益摘要、异常摘要、关键日志摘要。
- 风控日报至少包含规则命中次数、拒单统计、解除事件、严重告警摘要。
- 审计查询支持按用户、资源类型、动作、时间范围检索，并支持导出。

### 自动化测试

- 后端集成测试覆盖：认证、账户查询、下单撤单、风控拒单、回测提交、运行实例启动。
- 前端或 E2E 测试覆盖：登录、进入 Trade、下单、Risk 拒单、Strategy Lab 提交回测、运行实例查看。
- 对关键回归场景建立稳定测试数据和可重复环境，避免测试结果依赖外部随机输入。
- 将自动化测试纳入 CI 流程，至少区分快速检查与完整回归两档。

### 可观测性与运维

- 统一应用日志格式，保证前端错误、API 请求、队列任务、运行实例日志可按 `request_id` 或 `trace_id` 串联。
- 增加核心监控看板：API SLA、数据库连接、Redis 状态、队列积压、回测成功率、运行实例健康度、订单失败率。
- 为关键告警设置阈值和通知路径，如实例心跳中断、订单失败率飙升、回测任务堆积、数据库连接耗尽。
- 固化 Docker Compose 测试环境和生产部署步骤，明确环境变量、密钥来源、迁移顺序和服务健康检查。
- 输出回滚方案：代码回滚、数据库迁移回滚策略、配置回滚、禁用高风险功能开关。

### 稳定性与安全

- 对登录、下单、队列任务提交等入口增加限流和异常保护。
- Broker 凭证、刷新令牌、敏感环境变量必须加密或通过安全配置注入，禁止明文入库和明文日志输出。
- 完成最小安全检查：鉴权绕过、越权访问、敏感数据泄露、日志脱敏、依赖漏洞扫描。
- 完成最小性能压测：登录、账户查询、下单、订单推送、回测提交、实例状态查询。

## 关键接口、模型与事件

### REST API

- `GET /api/v1/notifications`
- `POST /api/v1/notifications/{notification_id}/read`
- `GET /api/v1/reports/accounts/daily`
- `GET /api/v1/reports/strategies/daily`
- `GET /api/v1/reports/risk/daily`
- `GET /api/v1/audit/logs`
- `GET /api/v1/audit/logs/export`
- `GET /api/v1/system/metrics-summary`

### WebSocket 与通知事件

- `system.notifications.<user_id>`：统一承载关键系统通知和告警摘要。
- 可选 `ops.alerts.admin`：管理员级运营告警聚合频道。

### 核心数据模型

- `Notification`：类型、级别、目标用户、来源、已读状态、投递结果。
- `DailyAccountReport`：账户净值、收益、持仓摘要、订单摘要。
- `DailyStrategyReport`：策略或实例运行摘要、收益、异常、告警。
- `DailyRiskReport`：规则命中、拒单统计、告警分布、解除记录。
- `AuditQueryResult`：资源、动作、操作者、时间、追踪 ID、摘要。

## 里程碑与依赖

### 里程碑 1：报表与通知

- 完成通知中心、日报查询与导出、审计查询。
- 产出：运营可用的可读数据入口。

### 里程碑 2：自动化测试与 CI

- 完成关键流程自动化测试并接入 CI。
- 产出：上线前可重复验证的质量门禁。

### 里程碑 3：监控、部署与安全收口

- 完成监控看板、部署文档、回滚方案、压测和安全检查。
- 产出：上线准备完成的工程体系。

### 关键依赖

- 依赖前四阶段所有关键链路和数据模型已稳定。
- 依赖 test 环境具备稳定联调入口，必要时可模拟外部 Broker 或行情。
- 若生产环境尚未完全就绪，仍需先完成 prod 配置与部署文档，不得只停留在 test 环境说明。

## 验收与测试场景

### 测试矩阵

| 类型 | 必测内容 |
| --- | --- |
| 单元测试 | 通知状态变更、报表聚合、审计筛选、限流与脱敏逻辑 |
| 集成测试 | 报表查询导出、通知已读、审计查询、系统指标汇总、关键 API 链路 |
| 联调 / E2E | 登录、下单、风险拒单、回测提交、运行实例查看、通知接收 |
| 人工验收 | 监控看板可读、告警路径可触达、按文档完成部署和回滚演练 |

### 核心验收场景

1. 管理员可在通知中心看到运行异常、风险告警和关键任务结果，并可标记已读。
2. 管理员可导出某日账户日报、策略日报和风控日报，数据与前四阶段事实表一致。
3. CI 在提交后自动执行关键测试，失败时阻断构建或发布流程。
4. 运维人员按部署文档可在 test 或 prod 环境完成部署、迁移和健康检查。
5. 模拟高错误率或实例心跳中断时，监控看板和通知路径能及时体现。

## 风险与回退方案

- 报表口径不一致：所有日报必须直接复用同一事实源和聚合规则，不允许前端二次拼接口径。
- 告警过载导致不可用：先定义关键告警优先级和收敛规则，避免把所有日志都变成通知。
- 自动化测试不稳定：优先固定数据、隔离外部依赖、区分 smoke 与 full regression。
- 部署流程依赖人工经验：必须将环境变量、迁移顺序、回滚步骤写成可执行文档。
- 安全治理滞后：未完成凭证保护、脱敏、基本越权检查前，不进入生产环境。

## 交付清单

- 通知中心、报表查询与导出、审计查询能力。
- 自动化测试套件、CI 质量门禁和回归场景集。
- 监控看板、告警规则、Trace ID 贯通方案。
- Docker 化部署文档、迁移和回滚说明。
- 压测结果、安全检查清单和上线前检查表。
