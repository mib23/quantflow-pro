# Phase 3：Strategy Lab 与回测体系

## 阶段目标

- 在前两阶段稳定的账户、订单、风险基础上，完成研究态策略管理和异步回测闭环。
- 让 `Strategy Lab` 从占位页面升级为可创建策略、管理版本、提交回测、查看结果的研究工作区。
- 将“策略编辑”与“策略执行”继续解耦，回测只通过后台任务和标准化结果模型落地。

## 进入条件与完成定义

### 进入条件

- Phase 1 已提供可复用的账户、订单、成交、行情接口和会话体系。
- Phase 2 已提供可复用的风控摘要、审计模型和事件记录方式。
- `server` 中 RQ 与 worker 基座可运行，`StrategyPage` 已保留研究域边界和页面落点。

### 完成定义

- 用户可创建策略、保存草稿、复制版本、查看版本历史。
- 用户可提交回测任务，任务进入队列、执行、完成或失败，并可查询状态。
- 回测结果可查看权益曲线、关键指标、交易明细，并支持下载报告。
- 回测链路具备任务日志、失败原因、重试控制和审计留痕。
- 本阶段不进入真实实盘调度，所有能力限定在研究态与回测态。

### 输出给下一阶段

- 稳定的策略元数据模型、版本模型、回测任务模型和结果模型。
- 可被 Phase 4 复用的策略部署输入，如版本 ID、参数快照、风险配置引用。
- Worker 任务模式、日志模型和任务状态流转规则，为运行实例调度提供模板。

## 范围与非目标

### 本阶段必做闭环

- 策略元数据管理：创建、编辑、复制、查看、归档。
- 策略版本管理：版本号、参数模板、代码快照、版本说明。
- 回测任务系统：提交、排队、执行、查询、取消或失败重试。
- 结果展示：权益曲线、指标概览、交易明细、报告下载。
- 审计与日志：任务提交、取消、重试、结果生成留痕。

### 可延后项

- 参数优化、网格搜索、Walk-forward、多资产组合回测。
- 在线协作编辑、多人评论、版本审批。
- 大规模分布式计算、GPU 加速、回测资源池自动扩缩容。
- 复杂因子分析、归因分析和研究 notebook 集成。

### 不在本阶段处理

- 策略仿真长期运行和实盘发布。
- 实盘审批流、运行心跳、故障恢复。
- 运营通知中心和生产级监控面板。

## 工作流拆解

### 前端

- 将 `StrategyPage` 从说明型页面改造为研究工作区，包含策略列表、版本列表、参数面板、任务列表和结果详情。
- 增加策略创建、编辑、复制版本、提交回测的表单与交互流。
- 结果页展示权益曲线、收益指标、风险指标、交易明细和报告下载入口。
- 为回测任务状态增加轮询或事件刷新能力，支持 `QUEUED`、`RUNNING`、`SUCCEEDED`、`FAILED`、`CANCELED`。
- 页面只消费标准化策略和回测 DTO，不直接依赖 worker 原始日志结构。

### 后端

- 建立 `strategies` 模块：策略元数据、策略版本、参数模板、归档状态、创建人与更新时间。
- 建立 `backtests` 模块：任务提交接口、任务查询接口、结果查询接口、报告下载接口。
- 回测任务提交时固化策略版本快照、参数快照、时间范围、标的集合、基准配置。
- 将回测执行委托给 RQ worker，API 层只负责校验、入队和状态查询。
- 回测结果至少包含：收益率、Sharpe、最大回撤、胜率、交易次数、权益曲线点位、交易明细摘要。
- 对回测失败进行标准化分类，如参数错误、数据缺失、运行异常、超时终止。

### 数据与集成

- 落地 `strategies`：策略基本信息、归属用户、状态、默认参数模板、最近版本引用。
- 落地 `strategy_versions`：版本号、代码快照或文件引用、参数快照、变更说明、创建时间。
- 落地 `backtest_jobs`：任务 ID、策略版本 ID、提交人、状态、队列信息、开始/结束时间、失败原因。
- 落地 `backtest_results`：任务 ID、指标摘要、权益曲线、交易明细引用、报告文件引用。
- 对策略代码或回测报告的存储方式做明确约束：小文件可先入库，大对象或导出文件建议走对象存储引用。
- 为回测使用的历史行情数据定义最小输入契约；如一期尚无全量行情仓储，可在 test 环境使用标准化样本数据集。

### Worker 与任务执行

- 约定任务 payload 至少包含：策略版本 ID、参数快照、回测窗口、执行环境标识、提交人。
- Worker 负责加载策略版本、准备数据、执行回测、保存结果、写入日志和状态。
- 对长任务设置超时、最大重试次数和显式取消检查点。
- 日志至少分为：任务日志、策略运行日志、错误堆栈摘要；前端默认展示结构化摘要，原始日志可后续扩展。
- 结果写入必须具备幂等保护，避免重复消费消息导致多份结果。

### 测试与运维

- 建立标准回测样本策略和样本数据，确保回归结果可预测。
- 为队列、worker、结果写入建立最小监控字段：任务数、失败数、平均耗时、超时数。
- 规定 test 环境的回测资源上限，避免单个任务拖垮共享环境。
- 将回测报告导出格式在本阶段固定为一到两种，如 JSON 摘要和 CSV/HTML 报告。

## 关键接口、模型与事件

### REST API

- `GET /api/v1/strategies`
- `POST /api/v1/strategies`
- `GET /api/v1/strategies/{strategy_id}`
- `POST /api/v1/strategies/{strategy_id}/versions`
- `POST /api/v1/strategies/{strategy_id}/versions/{version_id}/clone`
- `GET /api/v1/backtests`
- `POST /api/v1/backtests`
- `GET /api/v1/backtests/{job_id}`
- `POST /api/v1/backtests/{job_id}/cancel`
- `GET /api/v1/backtests/{job_id}/result`
- `GET /api/v1/backtests/{job_id}/report`

### 任务与事件

- 队列任务：`backtest.execute`
- 可选推送频道：`strategy.backtests.<user_id>`，用于任务状态变化通知。
- 审计事件：策略创建、版本复制、回测提交、取消、重试、报告导出。

### 核心数据模型

- `Strategy`：策略 ID、名称、描述、归属用户、状态、默认版本引用。
- `StrategyVersion`：版本 ID、版本号、代码快照、参数模板、变更说明。
- `BacktestJob`：任务 ID、版本 ID、状态、提交人、时间范围、失败原因。
- `BacktestResult`：收益指标、风险指标、权益曲线、交易明细、报告引用。

## 里程碑与依赖

### 里程碑 1：策略与版本管理

- 完成策略元数据、版本快照、复制版本和页面基础管理。
- 产出：可被回测与后续部署复用的稳定版本实体。

### 里程碑 2：回测任务入队与执行

- 完成任务提交、队列消费、状态流转、失败重试和取消。
- 产出：可追踪的异步回测链路。

### 里程碑 3：结果展示与导出

- 完成权益曲线、指标、交易明细、报告导出。
- 产出：研究态可验收的完整回测闭环。

### 关键依赖

- 依赖 Phase 1、Phase 2 已稳定的账户/订单/风险术语和审计模型。
- 依赖 RQ worker 基础设施可运行。
- 如真实行情历史仓储尚未完备，可先定义标准化样本数据输入，但接口和任务流转不可后改。

## 验收与测试场景

### 测试矩阵

| 类型 | 必测内容 |
| --- | --- |
| 单元测试 | 参数模板校验、任务状态流转、回测结果聚合、版本复制逻辑 |
| 集成测试 | 策略创建、版本保存、回测提交、任务查询、结果查询、报告导出 |
| 联调 / E2E | 在 Strategy Lab 创建策略、提交回测、查看结果、下载报告 |
| 人工验收 | 失败任务可见错误原因、取消任务不再继续写结果、版本历史可追溯 |

### 核心验收场景

1. 研究员创建一条新策略并保存版本 `v1`。
2. 用户基于 `v1` 提交回测任务，任务进入 `QUEUED` 后转为 `RUNNING`。
3. 回测完成后，页面可查看权益曲线、收益指标、交易明细和报告下载入口。
4. 用户复制 `v1` 生成 `v2`，两者参数快照和结果记录互不覆盖。
5. 回测失败时，前端可看到标准化错误原因，后端日志和审计记录可追踪。

## 风险与回退方案

- 回测输入数据不稳定导致结果不可比：为 test 环境固定样本数据集和期望结果。
- 任务执行时间过长：设置超时、取消点和资源上限，避免 worker 被长任务占满。
- 代码存储策略不清晰导致后续迁移困难：本阶段必须明确“入库快照”还是“对象存储引用”之一。
- 回测结果体量过大：指标摘要和曲线点位可入库，明细和报告可走对象存储引用。
- 如任务推送未及时落地，可先使用轮询查询状态，但任务状态模型必须保持稳定。
