# Phase 2：Dashboard 与 Risk 闭环

## 阶段目标

- 在 Phase 1 真实账户与订单链路的基础上，完成 Dashboard 去 mock 和 Risk 规则闭环。
- 让 Dashboard 成为真实账户、持仓、PnL、告警的聚合视图，而不是静态展示页。
- 让 Risk 页面从只读概念页升级为“规则可配置、可生效、可追踪”的管理闭环，并真正进入下单路径。

## 进入条件与完成定义

### 进入条件

- Phase 1 已提供稳定的账户总览、持仓、订单、成交、行情与订单状态事件。
- 前端已具备用户会话、查询层、WebSocket 基座和基础权限上下文。
- 后端已具备 `risk` 模块骨架和 `audit` 能力的落点。

### 完成定义

- Dashboard 的指标卡、持仓概览、告警摘要、净值曲线不再直接依赖 `shared/mocks`。
- 风控规则支持配置、启停、版本化、查看历史和审计追踪。
- 下单前风控预检查可返回通过或拒绝结果，拒单原因对前端和日志可见。
- 风险命中事件能写库、推送、查询，并在 Dashboard 与 Risk 页面联动展示。
- 关键风控动作和规则变更具备审计留痕。

### 输出给下一阶段

- 可复用的账户净值、风险摘要、告警事件接口，供 Strategy Lab 展示研究态指标。
- 统一的风险事件模型、审计模型和拒单错误码，为 Phase 3 回测与策略运行预留风控钩子。
- 稳定的 Dashboard 数据聚合方式，可在后续阶段扩展更多运营和监控指标。

## 范围与非目标

### 本阶段必做闭环

- Dashboard：账户总览、持仓统计、PnL、最近告警、净值曲线真实化。
- Risk：规则列表、规则详情、启停、版本、适用范围、命中事件查询。
- 下单风控：预检查接口、后端拒单逻辑、错误码映射、审计记录。
- 实时联动：风险命中事件推送到 Dashboard 告警区和 Risk 事件流。

### 可延后项

- 复杂多级审批、多人复核、人工处置工作流。
- 风控回放、批量规则导入导出、规则 DSL。
- 高级风险分析，如 VaR、因子暴露、关联账户敞口聚合。
- 更复杂的净值曲线回算和长期报表。

### 不在本阶段处理

- 策略版本、回测任务、研究文件管理。
- 仿真和实盘运行实例管理。
- 通知中心、报表中心、生产级监控看板。

## 工作流拆解

### 前端

- 将 Dashboard 页面中的账户概览、PnL、持仓列表、告警摘要改为查询真实聚合接口。
- 将净值曲线改为读取后端提供的时间序列接口，不再使用静态示例数据。
- 改造 `RiskPage`，支持规则列表、规则详情、启停状态、版本信息、最近命中事件。
- 在下单流程中增加预检查步骤：前端提交订单前调用 `/risk/checks/pre-trade`，将拒单原因清晰呈现给用户。
- 为 Dashboard 和 Risk 页接入 `risk.events.<account_id>` 频道，支持实时追加风险事件。
- 明确前端只读取标准化 risk DTO，不直接拼接订单和账户原始结构。

### 后端

- 扩展 `risk` 模块，定义规则实体、版本实体、作用范围、启停状态与基础校验逻辑。
- 完成预检查接口，根据账户总览、订单 notional、标的限制、交易时段等最小规则集返回判断结果。
- 在 `orders` 写链路中接入风控调用：若预检查失败，直接拒绝下单并返回 `ORDER_RISK_REJECTED`。
- 将规则变更、启停、命中、解除等动作写入 `audit_logs`。
- 提供 Dashboard 聚合接口，聚合账户净值、日内收益、持仓汇总、风险摘要、最近告警。
- 提供净值曲线或资产变化序列接口，支持指定时间窗口查询。
- 风险命中事件写入数据库后，广播到 WebSocket 频道，并同步更新 Dashboard 告警摘要。

### 数据与集成

- 落地 `risk_rules`，至少包含规则类型、阈值、适用账户、适用标的范围、状态、版本号、创建人与更新时间。
- 落地 `risk_events`，至少包含命中规则 ID、关联订单 ID、严重级别、命中原因、处置状态、发生时间。
- 扩展 `audit_logs`，至少记录操作人、动作、资源类型、资源 ID、变更前后摘要、追踪 ID。
- 为 Dashboard 净值曲线设计可持续扩展的存储方案，一期可以按日内采样快照或账户余额历史表实现。
- 规则执行必须依赖标准化 DTO，不允许 `risk` 模块直接耦合 Alpaca SDK。

### 测试与运维

- 建立风险规则种子数据，如单笔下单限额、日亏损阈值、受限标的清单、盘中交易时段限制。
- 对关键拒单场景保留可回放测试数据，方便回归验证。
- 日志必须包含规则 ID、账户 ID、订单 ID、命中原因和请求链路。
- 为 Dashboard 聚合查询设置合理缓存或预聚合策略，避免页面每次刷新触发过重查询。

## 关键接口、模型与事件

### REST API

- `GET /api/v1/dashboard/overview`
- `GET /api/v1/dashboard/equity-curve`
- `GET /api/v1/risk/summary`
- `GET /api/v1/risk/rules`
- `POST /api/v1/risk/rules`
- `PUT /api/v1/risk/rules/{rule_id}`
- `POST /api/v1/risk/rules/{rule_id}/activate`
- `POST /api/v1/risk/rules/{rule_id}/deactivate`
- `GET /api/v1/risk/events`
- `POST /api/v1/risk/checks/pre-trade`

### WebSocket 事件

- `risk.events.<account_id>`：推送规则命中、解除、熔断提示。
- `system.notifications.<user_id>`：本阶段可复用为关键拒单与风险告警提示。
- `orders.status.<account_id>`：继续作为风险联动时的下单状态输入。

### 核心数据模型

- `RiskRule`：规则类型、阈值、适用范围、状态、版本、描述。
- `RiskRuleVersion`：版本号、配置快照、变更原因、变更人、变更时间。
- `RiskEvent`：事件 ID、规则 ID、账户 ID、订单 ID、严重级别、状态、处置备注。
- `DashboardOverview`：账户总览、持仓统计、PnL、最近告警、实时健康摘要。

## 里程碑与依赖

### 里程碑 1：Dashboard 去 mock

- 完成真实聚合接口、净值曲线查询和前端真实展示。
- 产出：Dashboard 不再依赖静态示例数据。

### 里程碑 2：规则配置与查询

- 完成规则模型、列表、详情、启停、版本和事件查询。
- 产出：Risk 页面具备配置和追踪能力。

### 里程碑 3：下单前置风控

- 完成预检查、拒单错误码、前端提示和审计落库。
- 产出：风控正式进入交易闭环。

### 里程碑 4：实时风险事件流

- 完成风险事件写库、推送、Dashboard 和 Risk 联动展示。
- 产出：Phase 3 可复用的风险事件输入。

### 关键依赖

- 依赖 Phase 1 的真实订单和账户数据作为规则执行输入。
- 若净值曲线历史数据不足，可先按 test/local 环境生成最小采样快照，但接口形状必须稳定。
- 风险规则以账户级、标的级最小集合起步，不在本阶段引入复杂 DSL。

## 验收与测试场景

### 测试矩阵

| 类型 | 必测内容 |
| --- | --- |
| 单元测试 | 风控规则计算、阈值判断、规则版本快照、Dashboard 聚合映射 |
| 集成测试 | 规则创建、启停、预检查、拒单、事件查询、审计写入 |
| 联调 / E2E | Dashboard 加载真实概览、下单触发预检查、Risk 页面查看命中事件 |
| 人工验收 | 风险告警在 Dashboard 与 Risk 同步出现、禁用规则后不再触发拒单 |

### 核心验收场景

1. 管理员创建一条“单笔下单金额上限”规则并启用。
2. 交易员在 `Trade` 页面提交超限订单，系统返回 `ORDER_RISK_REJECTED` 且前端展示明确原因。
3. 拒单事件写入 `risk_events` 和 `audit_logs`，Risk 页面可查询，Dashboard 告警区可看到实时提醒。
4. 管理员停用该规则后，同等订单不再被该规则拦截。
5. Dashboard 的账户总览、PnL、净值曲线来自真实后端接口，而非 `shared/mocks`。

## 风险与回退方案

- 聚合查询过重导致 Dashboard 响应慢：先采用缓存和时间窗口限制，必要时预聚合。
- 风控规则误配导致全局拒单：规则启用前增加最小校验和 test 环境回放；保留快速停用能力。
- 风险事件过多影响页面性能：前端只展示最近窗口，历史查询走分页接口。
- 审计字段缺失导致无法追责：规则变更和拒单事件写入必须与主链路同事务或同追踪链路完成。
- 如部分风控逻辑尚未稳定，可先在配置中标记为只观察不阻断；但阻断型规则必须对用户返回清晰错误。
